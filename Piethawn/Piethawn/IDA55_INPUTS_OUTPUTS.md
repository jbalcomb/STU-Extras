# IDA55 Inputs And Outputs

This file describes the main artifacts used by the Piethawn IDA 5.5 synchronization workflow.

## Inputs

### Manual inputs

- `in/MAGIC.asm`
  - Full assembly listing exported manually from `MAGIC.idb`.
- `in/WIZARDS.asm`
  - Full assembly listing exported manually from `WIZARDS.idb`.
- `in/MAGIC.idb.ida55-functions.txt`
  - Text dump generated by `ida55_dump_functions.idc` from `MAGIC.idb`.
- `in/WIZARDS.idb.ida55-functions.txt`
  - Text dump generated by `ida55_dump_functions.idc` from `WIZARDS.idb`.
- `in/MAGIC.idb.ida55-function-colors.txt`
  - Text dump generated by `ida55_dump_function_colors.idc` from `MAGIC.idb`.
- `in/WIZARDS.idb.ida55-function-colors.txt`
  - Text dump generated by `ida55_dump_function_colors.idc` from `WIZARDS.idb`.
- `in/MAGIC.EXE`
  - Original executable used for DOS header and Borland overlay metadata extraction.
- `in/WIZARDS.EXE`
  - Original executable used for DOS header and Borland overlay metadata extraction.
- `project_config.json`
  - Stores the latest IDA database timestamp snapshot used for freshness checks.
- `ida55_exact_match_overrides.json`
  - Manual override map for cross-game segment pairing.
  - Interpretation:
    - each `magic_segment` / `wizards_segment` pair defines a positional equivalence domain
    - `pairing = "by_position"` means function 0 matches function 0, function 1 matches function 1, and so on

## Derived outputs

### EXE metadata

- `MAGIC.exe-header.json`
- `WIZARDS.exe-header.json`

Usage:

- inspect DOS `MZ` header fields
- inspect relocation counts
- inspect `e_lfanew` and secondary-header probe status

Interpretation:

- these files describe the executable container, not function matching
- they are useful for low-level validation and Borland-era executable structure analysis

### Borland overlay metadata

- `MAGIC.borland-ovrinfo.json`
- `WIZARDS.borland-ovrinfo.json`

Usage:

- inspect Borland `EXEINFO` / `SEGTABLE` data
- inspect embedded EXE name and `FILE_DATE`
- inspect overlay-related segment layout

Interpretation:

- these files explain how Borland arranged overlaid segments in the EXE
- they are useful context when mapping or validating overlay segments

### Split ASM tree

- `out/MAGIC/...`
- `out/WIZARDS/...`

Usage:

- review procedures by segment
- compare individual procedures as files
- feed normalized ASM comparison scripts

Interpretation:

- each segment becomes a directory
- each procedure becomes its own `.asm` file
- `_misc.asm` contains non-procedure lines from a segment

### Function dump JSON

- `in/MAGIC.idb.ida55-functions.json`
- `in/WIZARDS.idb.ida55-functions.json`

Usage:

- machine-code-first matching
- per-function byte hashing
- procedure size and chunk inspection

Interpretation:

- each function record includes `name`, `segment`, `start_ea`, `end_ea`, `bytes_hex`, and `bytes_sha256`
- exact `bytes_sha256` equality means exact machine-code equality
- mismatched hashes do not necessarily mean different logic if address operands changed

### Split ASM index and diff reports

- `out/asm-dedupe-index.json`
- `out/asm-diff-report/...`

Usage:

- normalized assembly comparison
- near-match review
- cross-checking text-level differences

Interpretation:

- these are heuristic, text-oriented analyses
- they are secondary to the machine-code matcher
- use them to understand naming drift, comment drift, or small local source differences

### Machine-code match report

- `out/ida55-function-match.json`

Usage:

- authoritative cross-game function pairing input
- rename IDC generation
- validation of exact byte matches versus positional overrides

Interpretation:

- `exact_byte_groups`
  - functions with identical machine-code hashes across games
- `position_override_pairs`
  - functions forced to match because the segment pair is manually asserted and paired by order
- `same_name_pairs`
  - same-name cross-game comparisons, useful for review
- `effective_exact_match`
  - true if the pair is exact by bytes or by manual positional override
- `override_segment_pair`
  - present when the match exists because of the manual segment map

### Rename IDC outputs

- `out/ida55-sync/magic_rename_to_wizards.idc`
- `out/ida55-sync/magic_rename_to_wizards.json`

Usage:

- apply `WIZARDS` function names to paired functions in `MAGIC.idb`
- review rename plan before running the IDC

Interpretation:

- the IDC script is the final automated output of the pipeline
- the JSON manifest lists each rename as `old_name -> new_name`
- only effective exact matches with different names are emitted

### Function color dump JSON

- `in/MAGIC.idb.ida55-function-colors.json`
- `in/WIZARDS.idb.ida55-function-colors.json`

Usage:

- inspect per-function IDA color state
- identify functions marked complete in either database
- feed the completion merge/sync workflow

Interpretation:

- each function record includes `item_color_hex`, `funcattr_color_hex`, and `has_user_color`
- the current completion workflow uses `00FFFF80` as the completed-transcription color
- default/unset functions use `FFFFFFFF`

### Completion merge outputs

- `out/ida55-sync/completion-merge.json`
- `out/ida55-sync/magic_sync_completion.idc`
- `out/ida55-sync/wizards_sync_completion.idc`

Usage:

- merge completion state across duplicate/equivalent functions
- apply completion color back into either IDB if either side says a function is complete
- audit synchronization gaps between `MAGIC` and `WIZARDS`

Interpretation:

- `components`
  - connected groups of equivalent functions built from exact-byte matches and positional override matches
- `is_completed`
  - true if any member of the component has the completed color
- `updates_by_game`
  - functions that still need the completed color applied in a specific IDB
- `sync_update_count_by_game`
  - if this is non-zero, the completion IDC scripts still need to be run

### Completion sync logs

- `MAGIC.idb.completion-sync.log`
- `WIZARDS.idb.completion-sync.log`

Usage:

- confirm whether IDA actually updated the requested functions
- troubleshoot missing symbols or failed color writes

Interpretation:

- `missing`
  - the IDC script could not find the function by name
- `already-set`
  - the function already had the desired completion color
- `setcolor-failed`
  - IDA rejected the color change
- `updated`
  - the function color was changed successfully

### IDA database timestamp config

- `project_config.json`

Usage:

- snapshot the current modified times of `MAGIC.idb/id0/id1` and `WIZARDS.idb/id0/id1`
- decide whether the dump/pipeline outputs are stale
- support scheduled freshness checks

Interpretation:

- `ida55_database_timestamps.databases.<GAME>.files`
  - per-file size and modified timestamps
- `ida55_database_timestamps.databases.<GAME>.latest_modified_utc`
  - the newest modified time across `idb`, `id0`, and `id1`
- if this timestamp changes, the corresponding dumps and downstream artifacts should be considered potentially stale

### Scheduled refresh task

- Windows Scheduled Task: `Piethawn IDA55 Timestamp Snapshot`

Usage:

- run the database timestamp snapshot automatically once per day

Interpretation:

- this task runs `ida55_db_timestamp_config.py`
- it updates `project_config.json` even if no manual dump pipeline is run that day
- it is only a freshness signal; it does not rerun the full analysis pipeline

### Pair-diff reports

- `out/Save_SAVE_GAM.pair-diff.md`
- `out/Load_SAVE_GAM.pair-diff.md`

Usage:

- inspect why two functions are or are not exact byte matches
- compare raw bytes, raw ASM, and normalized ASM in one report

Interpretation:

- useful for debugging disagreements between apparent logical similarity and machine-code hashes
- especially useful for library/helper address differences and overlay target differences

## Which Output To Trust First

For synchronization decisions, use this order:

1. `out/ida55-function-match.json`
2. `out/ida55-sync/magic_rename_to_wizards.json`
3. pair-diff reports for disputed matches
4. `out/asm-dedupe-index.json` and `out/asm-diff-report/...` for text-level context

The split-ASM analysis is valuable, but the machine-code match report is the primary source for IDA synchronization.
The main runner defaults to `in/` for these files, but also falls back to the repo root if a specific input has not been moved yet.

For completion tracking decisions, use this order:

1. `out/ida55-sync/completion-merge.json`
2. `MAGIC.idb.completion-sync.log` or `WIZARDS.idb.completion-sync.log`
3. `in/*.ida55-function-colors.json`

For freshness decisions, use:

1. `project_config.json`
2. the manual dump dates for files under `in/`
