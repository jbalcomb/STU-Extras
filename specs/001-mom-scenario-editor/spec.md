# Feature Specification: Master of Magic Scenario Editor

**Feature Branch**: `001-mom-scenario-editor`
**Created**: 2026-03-08
**Status**: Draft
**Input**: User description: "Build an application that lets people manufacture starting scenarios for Master of Magic (1994) and share them as game-save files with other members of the community. They should be able to change the game options and whatever game-data is generated by the New Game process. It should generate a world-map based on the user's parameters, but also allow for hand-editing the world-map in a visual editor. They should be able to arrange the tiles however they like, but it should highlight map square terrain that does not fit with the smoothing rules. The C++ code will be compiled by Emscripten to WASM and hosted on itch.io."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Generate and Export a Scenario (Priority: P1)

A scenario creator opens the application in their browser, adjusts world generation parameters (land mass proportion, terrain distribution, number of continents), clicks "Generate Map", reviews the resulting world on both Arcanus and Myrror planes, then exports the scenario as a .GAM save file they can load in Master of Magic or ReMoM.

**Why this priority**: This is the core value proposition — without map generation and .GAM export, the tool has no purpose. A user who can only generate and export already has a usable product.

**Independent Test**: Can be fully tested by generating a map with default parameters and verifying the exported .GAM file loads successfully in the game engine.

**Acceptance Scenarios**:

1. **Given** the user has opened the application, **When** they click "Generate Map" with default parameters, **Then** a world map is displayed showing terrain across both planes.
2. **Given** a generated world map is displayed, **When** the user clicks "Export .GAM", **Then** a valid 123,300-byte .GAM file is downloaded to their device.
3. **Given** the user has exported a .GAM file, **When** they load it in Master of Magic or ReMoM, **Then** the game starts with the generated world and settings intact.

---

### User Story 2 - Configure Game Options and Wizard Setup (Priority: P2)

A scenario creator configures the game parameters that would normally be set during the "New Game" process: difficulty level, magic intensity, land size, number of wizards, and each wizard's race, retorts, starting spells, and portrait. These settings are reflected in the exported .GAM file so the game starts with the chosen configuration.

**Why this priority**: Game options are the second most important customization — they define the scenario's challenge and flavor. Without wizard and option configuration, the tool produces only generic maps.

**Independent Test**: Can be tested by setting specific game options (e.g., Impossible difficulty, 5 wizards, specific races), exporting, and verifying the .GAM file contains those settings when loaded.

**Acceptance Scenarios**:

1. **Given** the application is open, **When** the user selects difficulty "Impossible" and magic level "Powerful", **Then** the scenario's game data reflects those choices.
2. **Given** the user is configuring wizards, **When** they set Wizard 1 to High Elf race with the Alchemy retort, **Then** that wizard's data in the exported scenario matches the configuration.
3. **Given** the user has configured 4 of 6 wizard slots, **When** they export, **Then** the .GAM file contains exactly 4 active wizards with the specified attributes and 2 empty slots.

---

### User Story 3 - Hand-Edit the World Map (Priority: P3)

A scenario creator uses painting tools to place individual terrain tiles, terrain specials (mineral deposits, nightshade, wild game), roads, and corruption on the world map. They can switch between Arcanus and Myrror planes and navigate the map by scrolling and zooming. Undo/redo supports iterative editing.

**Why this priority**: Hand-editing transforms the tool from a random generator into a precision scenario editor. Users who want to craft specific geographical challenges or recreate custom worlds need tile-level control.

**Independent Test**: Can be tested by painting a specific terrain pattern (e.g., a ring of mountains around a grassland valley), undoing several actions, then verifying the map state matches expectations.

**Acceptance Scenarios**:

1. **Given** a generated or blank world map, **When** the user selects the "Mountain" terrain tool and clicks on an ocean tile, **Then** that tile changes to mountain terrain and the display updates immediately.
2. **Given** the user has made 5 terrain edits, **When** they press undo 3 times, **Then** the map reverts to the state after the 2nd edit.
3. **Given** the user is viewing Arcanus, **When** they switch to Myrror, **Then** the Myrror plane map is displayed and all editing tools continue to function.

---

### User Story 4 - Place and Configure Entities (Priority: P4)

A scenario creator places cities, units, magic nodes, lairs, towers, and fortresses on the map. Each placed entity can be configured (e.g., a city's owner, race, population, and buildings; a unit's type and enchantments; a node's magic type). Entity placement respects slot limits (100 cities, 1009 units, 30 nodes, etc.).

**Why this priority**: Entities define the strategic landscape of a scenario — without cities, units, and nodes, maps are empty terrain with no gameplay.

**Independent Test**: Can be tested by placing a city for Wizard 1, adding units nearby, placing a node, then exporting and verifying these entities appear correctly in the loaded game.

**Acceptance Scenarios**:

1. **Given** the user selects the "Place City" tool, **When** they click on a grassland tile, **Then** a city is created at that location with default attributes that can be edited in a properties panel.
2. **Given** 100 cities have been placed, **When** the user attempts to place a 101st city, **Then** the system displays a message indicating the maximum city count has been reached.
3. **Given** a city has been placed, **When** the user selects it and changes its race to "Dwarf" and adds a Smithy building, **Then** those attributes persist in the scenario data and export.

---

### User Story 5 - Terrain Smoothing Rule Validation (Priority: P5)

While editing the map, the editor highlights tiles that violate terrain smoothing rules. Invalid adjacencies (e.g., shore tiles not adjacent to ocean, isolated river tiles, volcano next to tundra) are visually marked so the user can identify and fix them. The user may choose to ignore warnings — all tile arrangements are allowed, but warnings help produce game-compatible maps.

**Why this priority**: Smoothing validation is a quality-of-life feature that helps users produce maps that render correctly in-game. It enhances the editor but is not required for basic functionality.

**Independent Test**: Can be tested by placing a shore tile surrounded entirely by grassland (no adjacent ocean) and verifying it is highlighted as a smoothing violation.

**Acceptance Scenarios**:

1. **Given** a tile is placed that violates smoothing rules, **When** the map renders, **Then** that tile is visually distinguished (e.g., highlighted border or overlay) from conforming tiles.
2. **Given** the user fixes a highlighted tile by changing its terrain to a compatible type, **When** the map re-renders, **Then** the warning highlight is removed from that tile.
3. **Given** the user has a map with smoothing warnings, **When** they export the .GAM file, **Then** the export proceeds successfully — warnings are advisory, not blocking.
4. **Given** the user wants to review all warnings, **When** they activate a "Show Violations" view, **Then** all non-conforming tiles across both planes are listed or highlighted simultaneously.

---

### User Story 6 - Save and Load Native Scenario Format (Priority: P6)

A scenario creator saves their work-in-progress to the application's native format (which preserves editor metadata such as scenario title, author, description, and version) and loads it later to continue editing. This enables iterative scenario development across sessions.

**Why this priority**: Save/load is essential for iterative work, but since the application runs in a browser, the native format is a file downloaded to and uploaded from the user's device. This is lower priority than the core editing and export features.

**Independent Test**: Can be tested by creating a scenario, saving it, reloading it, and verifying all map data, entities, options, and metadata are preserved exactly.

**Acceptance Scenarios**:

1. **Given** a scenario with terrain edits, placed entities, and configured game options, **When** the user clicks "Save Scenario", **Then** a native-format file is downloaded containing all scenario data and metadata.
2. **Given** the user has a previously saved native-format file, **When** they load it into the application, **Then** the map, entities, game options, and metadata are fully restored.
3. **Given** the user loads a scenario, makes additional edits, and saves again, **Then** the new file reflects all accumulated changes.

---

### User Story 7 - Share Scenarios with the Community (Priority: P7)

A scenario creator exports their .GAM file and shares it with other Master of Magic players. The application is accessible to anyone via a web browser at its itch.io URL — no installation required. Shared .GAM files can be loaded directly by the game.

**Why this priority**: Community sharing is the ultimate goal, but the sharing mechanism is straightforward (users distribute .GAM files through their preferred channels — forums, Discord, itch.io comments). The application enables sharing by being freely accessible and producing standard game files.

**Independent Test**: Can be tested by having one user create and export a .GAM, sending it to another user, and verifying it loads in their game installation.

**Acceptance Scenarios**:

1. **Given** the application is hosted on itch.io, **When** a user navigates to the URL in any modern desktop browser, **Then** the application loads and is fully functional without installation.
2. **Given** a user exports a .GAM file, **When** another user places it in their Master of Magic save directory, **Then** the game recognizes and loads the scenario.

---

### Edge Cases

- What happens when the user's browser does not support WebAssembly? The application should display a clear compatibility message.
- How does the application handle loading a .GAM file that was not created by this editor (e.g., a mid-game save)? It should load what it can and warn about unsupported state.
- What happens when the user attempts to generate a map with extreme parameters (e.g., 100% ocean or 100% mountain)? The generator clamps land proportion to [5%, 95%] to ensure at least some ocean and land exist (required for shore placement and playability). The clamped value is reflected in the UI, and a status message informs the user when clamping occurs.
- What happens when the user places an entity on an invalid tile (e.g., a city on ocean)? The editor should allow it but may warn, since the game engine may handle it unpredictably.
- How does the editor handle the map's horizontal wrapping (the world is a cylinder — left edge connects to right edge)? Smoothing rules and map display must account for wrap-around adjacency.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST generate a world map across two planes (Arcanus and Myrror) based on user-configurable parameters including land proportion, terrain distribution, and continent count.
- **FR-002**: System MUST allow users to paint individual terrain tiles from the 12 available terrain types (Ocean, Shore, Grassland, Forest, Mountain, Desert, Swamp, Tundra, Hill, River, Volcano, Lake).
- **FR-003**: System MUST allow users to place terrain specials on tiles (Iron, Coal, Silver, Gold, Gems, Mithril, Adamantium, Quork Crystals, Crysx Crystals, Hunters Lodge, Wild Game, Nightshade).
- **FR-004**: System MUST allow users to toggle tile flags (Road, Enchanted Road, Corruption) on individual tiles.
- **FR-005**: System MUST allow users to configure game-level settings: difficulty (Intro through Impossible), magic intensity, land size, and active wizard count (1-6).
- **FR-006**: System MUST allow users to configure each wizard's attributes: name, race (14 races), portrait, retorts, and starting spells.
- **FR-007**: System MUST allow users to place and configure cities with attributes including owner, race, population, name, and constructed buildings (up to 36 building types).
- **FR-008**: System MUST allow users to place and configure units with attributes including type, owner, location, and status, within the 1009-unit slot limit.
- **FR-009**: System MUST allow users to place magic nodes (Sorcery, Nature, Chaos), lairs, towers, and fortresses within their respective slot limits (30 nodes, 102 lairs, 6 towers, 6 fortresses).
- **FR-010**: System MUST export scenarios as .GAM binary files that are exactly 123,300 bytes and loadable by the Master of Magic and ReMoM game engines.
- **FR-011**: System MUST save and load scenarios in a native editor format that preserves all map data, entity data, game settings, and editor metadata (title, author, description, version).
- **FR-012**: System MUST support undo and redo for all map and entity editing operations.
- **FR-013**: System MUST visually highlight tiles that violate terrain smoothing rules, indicating which adjacency constraints are broken.
- **FR-014**: System MUST allow switching between Arcanus and Myrror plane views during editing.
- **FR-015**: System MUST allow map navigation through scrolling and zooming.
- **FR-016**: System MUST run entirely in the user's web browser without requiring server-side processing or installation.
- **FR-017**: System MUST provide file download for .GAM export and native format save, and file upload for native format loading, since there is no server-side storage.
- **FR-018**: System MUST display a minimap showing the full world overview and the user's current viewport position.
- **FR-019**: System MUST enforce entity slot limits and inform the user when a limit is reached.
- **FR-020**: System MUST account for horizontal map wrapping (cylindrical world) in map display, editing, and smoothing rule checks.

### Key Entities

- **Scenario**: Top-level container holding all world data, entity data, game settings, and editor metadata. Exists in two formats: native (for iterative editing) and .GAM (for game engine consumption).
- **World Map**: A grid of 60x40 tiles across two planes (Arcanus and Myrror), where each tile holds terrain type, terrain special, flags, landmass ID, and exploration state.
- **Tile**: Individual map cell with a base terrain type (one of 12), optional terrain special (mineral deposit or flora), and optional flags (roads, corruption).
- **Wizard**: A player or AI participant (up to 6 slots) with name, race, portrait, retorts, starting spells, and fortress location.
- **City**: A settlement (up to 100 slots) with owner, race, name, location, population, and constructed buildings.
- **Unit**: A military or hero unit (up to 1009 slots) with type, owner, location, and status. Units with type < 35 are heroes.
- **Magic Node**: A source of magical power (up to 30 slots) typed as Sorcery, Nature, or Chaos, with location and guardian information.
- **Lair**: A monster den or ruin (up to 102 slots) with location and contents.
- **Tower/Fortress**: Special structures (6 towers, 6 fortresses) linking planes or marking wizard seats of power.
- **Item**: A magical artifact (up to 138 slots) with attributes and enchantments.
- **Game Options**: Global settings governing difficulty, magic intensity, land size, turn number, and wizard count.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can go from opening the application to downloading a valid, game-loadable .GAM file in under 5 minutes using only default settings and generated terrain.
- **SC-002**: The application loads and becomes interactive within 10 seconds on a standard broadband connection (10 Mbps+).
- **SC-003**: Users can paint terrain at interactive frame rates (minimum 30 frames per second) while editing a full 60x40 map with entities placed.
- **SC-004**: 100% of exported .GAM files are exactly 123,300 bytes and load without error in ReMoM.
- **SC-005**: Smoothing rule violations are highlighted within 1 second of a tile edit, providing immediate visual feedback.
- **SC-006**: All entity slot limits are enforced with zero data corruption — placing an entity beyond its limit never overwrites existing data.
- **SC-007**: Save/load round-trip preserves 100% of scenario data — no information loss between save, close, and reload.
- **SC-008**: The application functions correctly in current versions of Chrome, Firefox, Edge, and Safari on desktop without plugins or extensions.
- **SC-009**: Users can complete a full scenario editing workflow (generate map, configure wizards, place entities, export) without consulting external documentation, using only in-application controls and labels.

## Assumptions

- The application is a single-page web application with no backend — all processing occurs client-side in the browser.
- Community sharing is handled externally by users (via forums, Discord, file hosting, itch.io comments) — the application itself does not host or index shared scenarios.
- The terrain smoothing rules are derived from Master of Magic's original terrain rendering logic, specifically the tile adjacency constraints that produce correct autotiling in the game engine.
- Map generation will use procedural algorithms (noise-based terrain generation, landmass shaping) parameterized by user inputs rather than importing external map data.
- The application assumes the user's browser supports WebAssembly and either WebGL or Canvas 2D rendering.
- itch.io hosting provides only static file serving — no server-side APIs, databases, or authentication are required.
- The original Master of Magic game data (sprites, palettes) is not bundled; the editor uses its own visual representation of terrain and entities.
